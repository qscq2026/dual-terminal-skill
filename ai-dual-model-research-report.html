一、研究背景与问题定义

1.1 核心需求

用户希望在基于 Pi 开源项目（Claude Code 的终端编码智能体）的基础上，实现双模型工作模式：

  • Worker（工作者）：负责生成代码、执行文件操作、完成具体任务
  • Verifier（验证者）：负责独立审查 Worker 的输出，检查流程合规性和代码质量

1.2 关键挑战

双模型架构的核心难点不在于"如何分工"，而在于如何确保 Verifier 的独立性。如果 Worker 和 Verifier 运行在同一个信任域内，Worker 可能：

  • 伪造审查结果，声称已审查通过
  • 跳过审查流程，直接输出
  • 篡改审查结果，将 NEEDS_FIX 改为 APPROVED
  • 利用模型权重的相似性产生自我偏袒
研究目标：在 Claude Code 生态内，找到一种既能保持开发效率，又能确保 Worker 无法影响 Verifier 判断的协作架构。

二、方案演进路线

本研究通过多轮探讨，逐步深化对隔离机制的理解，最终形成五种方案：

三、方案一：单模型 Skill 方案

3.1 架构

用户请求 │ ▼ ┌─────────────────┐ │ Claude Code │ ← 同一实例 │ Worker + │ 既生成又审查 │ Verifier │ └─────────────────┘ │ ▼ 输出结果

3.2 实现

通过单个 SKILL.md 文件定义流程规范，Claude 在对话中先扮演 Worker 生成代码，再切换角色扮演 Verifier 审查。

3.3 问题

  • 自我偏袒：Claude 对自己生成的代码难以保持客观批判
  • 上下文污染：Worker 的生成思路会影响 Verifier 的判断
  • 无强制约束：Claude 可以选择性执行审查步骤
结论：单模型方案在概念验证阶段可行，但无法解决"自我审查"的根本矛盾，不适合生产环境。

四、方案二：MCP Server 方案

4.1 架构

用户请求 │ ▼ ┌─────────────────┐ │ Claude Code │ ← Worker（本地） │ (Worker) │ └────────┬────────┘ │ 调用 MCP Tool ▼ ┌─────────────────┐ │ MCP Server │ ← Node.js/Python 进程 │ (Verifier) │ 调用第三方模型 API └────────┬────────┘ │ 返回审查结果 ▼ ┌─────────────────┐ │ Claude Code │ ← 裁决输出 │ (Arbiter) │ └─────────────────┘

4.2 实现

开发一个 MCP Server（约 50 行 Node.js），通过 claude mcp add 注册到 Claude Code。Worker 完成任务后，通过 mcp__verifier__review 工具调用外部模型审查。

4.3 优点

  • Verifier 运行在独立进程中，Worker 无法直接篡改其输出
  • 可以调用任意第三方模型（DeepSeek、GPT-4o、Gemini）
  • Claude Code 原生支持，集成度高

4.4 缺点

  • 需要开发 MCP Server，引入 Node.js/Python 依赖
  • Claude 作为 Arbiter 仍可能偏袒 Worker
  • 调试 MCP 工具较为复杂

五、方案三：外部脚本方案

5.1 架构

用户请求 │ ▼ ┌─────────────────┐ │ Claude Code │ ← Arbiter + Verifier │ (本地) │ └────────┬────────┘ │ bash 调用 ▼ ┌─────────────────┐ │ worker.sh │ ← Bash 脚本 │ (curl API) │ └────────┬────────┘ │ HTTP 请求 ▼ ┌─────────────────┐ │ 第三方模型 │ ← DeepSeek/GPT-4o │ (Worker) │ └─────────────────┘

5.2 实现

在 Skill 目录下创建 scripts/worker.sh，使用 curl 调用第三方模型 API。Claude 作为本地 Verifier 审查返回结果。

# 调用 Worker
bash .claude/skills/reviewer-worker/scripts/worker.sh \
    /tmp/rw-task.txt /tmp/rw-context.txt

# Worker 返回结果 -> Claude 审查
# Claude 输出 VERIFIER_REPORT

5.3 优点

  • 极简实现，仅需一个 bash 脚本（约 100 行）
  • 零额外依赖（系统自带 curl + python3）
  • Worker 是第三方模型，无法影响本地 Claude

5.4 缺点

  • Worker 只能生成文本，无法使用 Claude Code 的工具（Read/Edit/Bash）
  • 上下文传递有限，复杂项目理解不足
  • Claude 作为 Arbiter 仍可能偏袒

5.5 防作弊分析

作弊方式可行性原因
伪造审查结果可能Claude 作为 Arbiter，理论上可伪造
跳过审查可能Claude 可选择不执行脚本
篡改审查结果可能脚本返回后 Claude 可修改
Worker 自我偏袒不可能Worker 是第三方模型，不接触 Claude

六、方案四：Subagent 方案

6.1 架构

用户请求 │ ▼ ┌─────────────────────────────────────────┐ │ Claude Code (主代理) │ │ Arbiter │ │ 只调度，不生成 │ └──────────────┬──────────────────────────┘ │ ┌───────┴───────┐ ▼ ▼ ┌─────────────┐ ┌─────────────┐ │ Worker │ │ Verifier │ │ Subagent │ │ Subagent │ │ │ │ │ │ 独立上下文 │ │ 独立上下文 │ │ 可使用工具 │ │ 可使用工具 │ │ 可编辑文件 │ │ 只读权限 │ └─────────────┘ └─────────────┘

6.2 实现

在 .claude/agents/ 目录下创建两个 Subagent 定义文件：

#.claude/agents/worker.md
---
name: worker
isolation: worktree
model: claude-sonnet-4
tools: [Read, Edit, Write, Bash]
---

#.claude/agents/verifier.md
---
name: verifier
isolation: worktree
model: claude-opus-4
tools: [Read, Grep, Glob]
disallowedTools: [Write, Edit, Bash]
---

6.3 关键特性

  • 独立上下文窗口：Subagent 不继承父代理的对话历史
  • 独立系统提示：可专门化行为（Worker 生成，Verifier 审查）
  • Worktree 隔离：文件系统隔离，防止信息泄露
  • 工具权限分离：Verifier 可限制为只读

6.4 隔离性真相

隔离维度实际情况可信度
上下文窗口✅ 完全隔离高
系统提示✅ 完全隔离高
工具权限✅ 可独立配置高
文件系统⚠️ 默认共享，可用 worktree 隔离中
进程/内存❌ 不隔离，同一 Claude Code 进程低
模型实例❌ 不隔离，同一模型实例低

6.5 防作弊分析

作弊方式可行性原因
伪造审查结果不可能Worker 看不到 Verifier 的上下文
跳过审查不可能父代理控制调用时机
篡改审查结果不可能Verifier 直接返回给父代理
通过文件系统泄露可能默认共享文件系统
模型权重相似性偏袒微妙同一模型实例，行为模式相似
核心风险：Subagent 的隔离是上下文级别的，不是进程/实例级别的。Worker 和 Verifier 共享底层模型权重，可能存在微妙的"自我偏袒"倾向。

七、方案五：双终端方案

7.1 架构

终端 A (Worker) 终端 B (Verifier) │ │ │ 读取任务 -> 生成代码 │ │ 写入共享文件 │ │ │ │ 读取文件 -> 独立审查 │ │ │ 写入审查报告 │ │ │ 读取报告 <- │ │ 修正代码 -> 重新提交 │ │ │ +----------- 循环直到 APPROVED ----+ │ ▼ .dual-claude/ (共享目录) ├── status.txt ← 状态机 ├── task.txt ← 任务描述 ├── worker-output.txt ← Worker 输出 ├── verifier-report.txt ← 审查报告 └── iteration.txt ← 迭代计数

7.2 实现

两个完全独立的 Claude Code 实例，通过文件系统状态机协调：

  1. 初始化：bash scripts/start-dual-terminal.sh "任务描述"
  2. 终端 A：启动 Claude Code，加载 Worker Skill，生成代码
  3. 终端 B：启动 Claude Code，加载 Verifier Skill，独立审查
  4. 状态机：通过 .dual-claude/status.txt 强制等待
  5. 迭代：最多 3 次，超限人工介入

8.2 信任链分析

信任链长度与安全性关系 ═══════════════════════════════════════════════════════════ 单模型: 用户 ──→ Claude(Worker+Verifier) ← 1个信任点，最弱 │ └── 自我审查，无外部验证 外部脚本: 用户 ──→ Claude(Arbiter) ──→ 第三方API(Worker) │ ↑ └── Arbiter 可作弊，Worker 不可 Subagent: 用户 ──→ Claude(Arbiter) ──→ Worker Subagent │ └──→ Verifier Subagent │ └── Arbiter 可偏袒，但 Worker/Verifier 隔离 双终端: 用户 ──→ 终端A(Worker) ←──→ 终端B(Verifier) │ └── 两个独立实例，无共同信任点

8.3 终极防御：跨系统隔离

如果双终端仍使用同一个 Claude 模型，存在微妙的权重相似性风险。终极方案是：

  • 终端 A：Claude Sonnet 4（Worker）
  • 终端 B：GPT-4o via API（Verifier）— 完全不同的训练数据、架构、厂商

此时两个模型来自不同公司、不同架构、不同训练数据，不存在任何共享的底层表示，彻底消除偏袒可能。

十二、结论与展望

12.1 核心结论

  1. 隔离级别决定可信度：从上下文隔离 → 进程隔离 → 跨系统隔离，每提升一级，Worker 作弊的可能性降低一个数量级。
  2. 自动化与可信度是权衡：Subagent 自动化程度高但隔离性有限；双终端隔离性最强但需要人工协调状态转换。
  3. Claude 更适合做 Verifier 而非 Worker：Claude 的系统提示鼓励质疑和安全优先，作为 Verifier 能发挥批判性思维优势。
  4. 文件系统状态机是简单有效的协调机制：不需要复杂协议，通过文本文件的状态转换即可实现强制等待和审计追踪。

12.2 未来展望

  • 原生多代理支持：Claude Code 未来可能内置多代理协作机制，提供更优雅的调度方式
  • 自动评分器（Grader）：2026 年已支持 Performance Outcomes，未来可实现全自动迭代修正
  • 区块链审计：将审查日志写入不可篡改的存储，实现事后可追溯
  • 联邦审查：多个不同厂商的 Verifier 同时审查，多数投票决定结果

12.3 最终建议

对于大多数团队：使用 Subagent 方案，在自动化和可信度之间取得最佳平衡。

对于安全核心代码：使用 双终端 + 跨系统隔离（Claude Worker + GPT-4o Verifier），确保绝对不可作弊。

对于快速验证：使用 外部脚本方案，5 分钟即可部署，成本极低。
