## 二、状态机流转分析

```
IDLE → WORKER_DONE → NEEDS_FIX → WORKER_DONE → NEEDS_FIX → ... → APPROVED
```

**实际流转：**
| 路径 | 比例 | 说明 |
|------|------|------|
| `WORKER_DONE → NEEDS_FIX` | ~95% | 绝大多数都发现问题 |
| `WORKER_DONE → APPROVED` | ~2% | 仅v1.0终审、v2.0终审、v3.x终审 |
| `WORKER_DONE → REJECTED` | 0% | 未出现驳回 |

**问题：** 从未使用过 `REJECTED` 状态。当Worker反复修不好时，应该用 `REJECTED` 打断循环重启。实际上发生了 **2次Worker声称已修复但代码没改** 的情况，REJECTED本可以更早触发闭环。

| 对Verifier反馈响应不及时 | 多次 | monitor没有及时触发，需要用户多次提醒"看反馈报告" |

### ⚠️ 特别问题：Monitor/监控纪律

**最严重的流程违规：** 多次设置了后台监控但未正确等待结果，导致用户反复提醒"看反馈报告"。

| 轮次 | 问题 |
|------|------|
| 第15轮后 | 设置wait-for-status后未持续等待，用户提醒 |
| 多次循环 | 用Bash后台sleep 10而不是Monitor工具，输出不主动推送 |
| 第50-60轮 | 多次使用纯Bash轮询而非Monitor，导致通知延迟2分钟 |

**改进方案：** 状态监控必须用 `Monitor` 工具（每行输出推送给模型），不能用 `Bash run_in_background + sleep`。

## 五、流程中的关键断裂点

### 断裂1：代码改没改不知道（第45轮）

```
Worker执行Node.js替换脚本 → 脚本失败了没报错 →
Worker认为"已修" → 提交WORKER_DONE →
Verifier审代码发现return还在 → NEEDS_FIX

根因：Node的replace模式在CRLF/LF混用环境下匹配失败
教训：提交前必须做 `node -c` + `grep -n "关键词"` 双重确认
```

### 断裂2：Monitor不推消息（第50-60轮反复出现）

```
Bash run_in_background → sleep 10轮询 → 输出到文件 →
系统不推送通知 → 用户手动查看才发现状态已变更 →
用户暴怒"你TMD看看反馈报告"

根因：用了 Bash 而不是 Monitor 工具
教训：状态监控必须用 Monitor 工具
```

### 断裂3：Verifier没实测就通过（第43/48轮）

```
Verifier审查代码 → 看到overflow-x:auto加上了 → "通过" →
实际浏览器开375px视口 → 内容被overflow:hidden裁掉 →
第48轮才发现垂直溢出

根因：Verifier没有开浏览器测
教训：布局变更必须浏览器实测，不能只看CSS代码
```

## 六、流程建议

### 对Worker

| 建议 | 优先级 |
|------|--------|
| 提交前做 `node -c` + `grep` 双重验证代码更改 | P0 |
| 状态监控必须用 `Monitor` 工具 | P0 |
| UI/布局修改必须浏览器截图实测后再提交 | P1 |
| 诚实申报"我没把握"比硬撑过关代价小 | P1 |
| 涉及尺寸/数值的修改，先在注释里写下数学验算 | P2 |

### 对Verifier

| 建议 | 优先级 |
|------|--------|
| 布局/CSS修改必须浏览器实测，不能只看代码 | P0 |
| "代码没变""报告与代码不符"直接 REJECTED | P0 |
| 数学验算必须写在报告中（如33×22-10=374） | P1 |
| 同一个bug连续2轮没修好 → REJECTED触发重启 | P1 |
| 状态改为APPROVED前做最小回归测试 | P2 |

### 对流程本身

| 建议 | 优先级 |
|--------|--------|
| 增加第3个状态 `REJECTED` 的实际使用，避免无效循环 | P0 |
| 55轮后引入"已完成功能清单"对照审查，避免边际递减 | P1 |
| 每10轮做一次"这10轮修了什么"汇总，用于决定是否终审 | P1 |
| 状态监控统一用 Monitor，禁止用 Bash run_in_background sleep | P0 |

## 七、总结

**总体评价：**

双终端协作模式证明有效——71轮迭代从"几乎不能玩"到"体验不错"，10+个阻塞级bug全部修复，4301行代码质量稳定。但流程执行质量中后期有明显疲劳迹象。

**最严重的3个问题：**

1. **监控纪律**（多次不主动看反馈，用户反复提醒）——这是最影响用户信任的
2. **验证深度**（提交前不double check代码改了没，Verifier不实测就通过）
3. **REJECTED状态未被使用**（没有触发过重启循环，拖延了收敛速度）

**最成功的3件事：**

1. **根因三层问**（不止于表面，追到底层原因）
2. **数学验算**（精确到像素的布局计算）
3. **自我反省机制**（Verifier承认漏判并记录，形成改进闭环）

> "真正的工程开发不是写代码，是修改代码。" —— 71轮迭代的真实写照
